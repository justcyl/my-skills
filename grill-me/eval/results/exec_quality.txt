## 我对这个方案的理解

**目标**
解决 15 人研发团队文档分散（Notion、飞书、本地 Markdown、Confluence）导致的信息检索困难问题，两个月内上线一个内网跨平台聚合搜索工具。

**方案概要**
- 数据层：从各平台拉取/同步文档数据，写入 Elasticsearch 建立统一索引
- 存储层：MinIO 存附件（图片、文件等二进制资源）
- 服务层：Go + Gin 提供搜索 API，负责与各平台对接、数据同步、索引管理
- 展示层：React + Ant Design 前端，用户统一在一个界面搜索
- 部署：内网服务器，不出公司网络

**已确定的决策**
- 技术栈全部确定（ES + MinIO + Go/Gin + React/AntD）
- 两个月上线时间盒
- 内网部署（合规或安全要求）

**我的推断**（请纠正）
- 推断这是一个"搜索"工具而非"知识管理"工具——即只聚合搜索，不打算做统一编辑/写入
- 推断数据同步是单向的（从各平台拉到 ES），不打算双向写回
- 推断团队有 Go/React 能力，技术栈是现有能力选择而非新学
- 推断这个项目由一两个人在业务间隙兼职做，不是专职项目

**我看到的风险**（先亮出来）

1. **各平台 API 接入是最大的工时黑洞**：Notion API、飞书 API、Confluence API 各不相同，认证方式、频率限制、数据结构都要单独处理。"本地 Markdown 文件"甚至没有 API，需要约定目录同步机制。两个月内做完四个数据源接入 + 搜索本身，进度很紧。
2. **没有说清楚"搜索"的深度**：是标题搜索？全文搜索？要支持中文分词（IK Analyzer）吗？搜索结果点击后跳转原链接，还是在工具内渲染全文？
3. **MinIO 的必要性存疑**：如果只是聚合搜索跳转原平台，为什么需要存附件？除非要做离线缓存或全文索引 PDF/Word，否则 MinIO 是额外复杂度。
4. **数据新鲜度问题**：各平台文档更新后多久同步到 ES？实时 Webhook？定时轮询？飞书/Confluence 可以 Webhook，但本地 Markdown 怎么触发同步？
5. **"两个月上线"定义模糊**：上线是指可用？还是所有四个平台都接入完？如果两个月内只接了两个平台，算成功还是失败？

## 批量追问：第一批

> 这批问题围绕**范围与成功标准**——回答直接影响方案的可行性判断。

---

**Q1. 谁来开发？时间投入是多少？**
影响：这直接决定两个月能做多少事。
我的推断：1-2 人兼职。
选项：A）1 人全职 B）1-2 人兼职（~30% 时间）C）有专职外援

---

**Q2. 四个数据源必须同时上线，还是可以分期？**
影响：分期的话两个月完全可行，全量上线风险很高。
我的推荐：**强烈建议分期**，先上飞书或 Confluence 一个，验证搜索体验，再逐步接入。
选项：A）全部四个平台必须 Day 1 上线 B）可以分期，先做高优先级平台 C）还没想清楚

---

**Q3. 搜索结果点击后的期望行为是什么？**
影响：决定是否需要 MinIO，以及前端复杂度。
选项：A）跳转原平台链接（最简单）B）在工具内展示全文预览（需要内容缓存）C）离线全量存储，脱离原平台（需要 MinIO + 全文缓存）

---

**Q4. 本地 Markdown 文件如何同步？**
影响：本地文件没有 API，需要专门的同步机制，这是四个源里最麻烦的。
选项：A）手动上传/推送到指定目录 B）Git 仓库 Webhook 触发同步 C）文件系统监听（inotify 等）D）把 Markdown 直接迁移到某个平台，不做本地文件支持

---

**Q5. 这个项目的真实动机是什么——是解决搜索问题，还是借机统一迁移文档平台？**
影响：如果只是搜索，聚合方案够用；如果想趁机整合，方案会大很多。
我的推断：纯粹解决搜索痛点。
选项：A）只解决搜索，不迁移 B）搜索是切入点，长期想统一到一个平台 C）还没有长期规划
