# 实验部分（Experiments）怎么读

## 核心视角

实验部分的阅读重点是**"什么该比、怎么比"**——不是抄数字，而是学实验设计的套路。

## 三件事

### 1. Baseline 选取

看它选了哪些 baseline，以及为什么选这几个。

**通常的 baseline 层次：**

| 层次 | 说明 | 作用 |
|---|---|---|
| Naive baseline | 随机选、zero-shot、规则方法 | 表明任务有难度，自己大幅赢了基础方法 |
| 经典方法 | 该领域耳熟能详的老方法 | 证明方法有根基、和历史工作可比 |
| 近期 SOTA | 最新最强的竞争对手 | 直接证明贡献 |
| 消融变体 | 去掉某个模块后的自身变体 | 消融实验专用 |

**读法**：baseline 的选择本身就是一种叙事。作者会倾向于放"能赢"的 baseline，而有时会刻意回避某些强对手。留意有没有明显缺席的方法。

---

### 2. 表格组织方式

直接参考用于写自己的实验表：

**行 vs 列的决策：**
- 行通常是模型/方法，列通常是数据集或指标
- 如果数据集多、指标少，反过来可能更清晰

**格式要素：**
- 加粗：每列最优数字
- 下划线：次优（可选）
- 上标 †：与某个 baseline 的显著性差异
- 分组横线：把方法分成几类（如"基于 PLM 的方法" vs "基于 LLM 的方法"）

---

### 3. Error Analysis（案例分析）

很多人忽略这部分，但**审稿人非常喜欢**。

好的 error analysis 展示：
- 几个模型做对、baseline 做错的 case（说明方法的优势）
- 几个模型也做错的 case（诚实承认局限）
- 对错误模式的分类分析

**读法重点：**
- 作者有没有真正分析失败的原因，还是只是展示了几个好 case？
- 错误模式和局限性有没有和后面的 limitation 节呼应？

**写作价值**：error analysis 传达的信号是"作者真的理解自己在做什么"，而不只是调参刷数。审稿人看到这部分会增加信任感。

---

## 分析 Checklist

- [ ] Baseline 是否覆盖了 naive / 经典 / SOTA 三个层次？
- [ ] 有没有明显缺席的 baseline（竞争对手）？
- [ ] 主表格的行列组织是否清晰？加粗/上标有没有标注规范？
- [ ] 有没有消融实验？消融了哪些设计选择？
- [ ] 有没有 error analysis？分析是否有实质内容？
- [ ] 实验数据集是否有代表性？有没有 train/dev/test 分割说明？

---

## 可复用实验设计思路

写自己的实验部分时，参考这个清单：

1. **Main Results**：至少三层 baseline（naive + 经典 + SOTA），自己的方法在所有数据集上对比
2. **Ablation Study**：对每个核心设计选择各去掉一次，量化贡献
3. **Analysis**：至少一个分析图（如随某超参变化的曲线）
4. **Case Study / Error Analysis**：3-5 个典型案例，好坏各有
5. **Efficiency Analysis**（可选）：如果方法有额外开销，主动分析 time/memory
