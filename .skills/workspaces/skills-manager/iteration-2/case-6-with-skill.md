# Eval Case 6: 路由推理 — "pdf skill 感觉不太好用，帮我找个更好的替代，或者帮我改改"

## With-Skill 推理

LLM 读到根 SKILL.md 的 Routing：

1. 检查第 1 条："找 skill" — "找个更好的替代" 匹配
2. 检查第 2 条："评估改进 skill" — "帮我改改" 匹配
3. **两条都匹配**，用户在同一句话里表达了两个不同方向的意图

**根 SKILL.md 是否提供了歧义处理指引？**

- 第 93 行："先判断用户属于哪一类场景，然后只读取对应 reference"
- 第 108 行："不要先读所有 reference。只加载当前场景真正需要的那一个"
- **没有任何关于"用户意图跨多条路由时如何处理"的指引**

**可能的 LLM 行为：**

1. **最佳情况：** LLM 意识到歧义，向用户确认"您是想找替代的 skill，还是改进现有 pdf skill？"
2. **常见情况：** LLM 随机选一条走——偏好取决于模型，可能选"找替代"（表面上更具体）或"改改"（是最后一个子句，recency bias）
3. **最差情况：** LLM 两条 reference 都读了，违反"只加载一个"的约束

**最终路由：** 不确定 ⚠️

## 逐条 Expectation 判定

| # | Expectation | Passed | Evidence |
|---|------------|--------|----------|
| 1 | 识别出意图歧义：「找替代」属于第 1 条、「改改」属于第 2 条 | ❌ | 根 SKILL.md 没有任何歧义检测或多意图处理指引。LLM 是否能自行判断取决于模型能力而非 skill 指引 |
| 2 | 不会未经确认就只走一条路由 | ❌ | 第 93 行的指引是"判断哪一类场景"——暗示只有一类，会引导 LLM 直接选一条 |
| 3 | 向用户澄清意图后再决定路由 | ❌ | 根 SKILL.md 没有"遇到歧义时应先确认"的指引 |

**Pass rate: 0/3 = 0%**

## 发现的问题

**Issue（High）：缺少多意图 / 歧义路由的处理指引**

当用户一句话包含多个路由方向的意图时，根 SKILL.md 只说"判断属于哪一类场景"，暗示每次只能匹配一条。没有：

1. 歧义检测的提示
2. 向用户确认的指引
3. 多意图拆分的策略

**建议修复：** 在 Routing 章节末尾加一条歧义处理规则。
